# AGENTS.md

AIを使った開発における、チーム共通のエージェントルール。
全員が同じ基準で作業できることを目的とする。

---

## 1. 動作の基本原則

- 指示されたことだけを実行する（関係のないファイルは変更しない）
- 大きな変更の前には必ず確認を取る
- 情報が不足している場合は推測で進めず、質問する
- 1タスク＝1変更を基本とし、変更範囲を小さく保つ

---

## 2. 実装前の確認・計画（最重要）

### 基本方針
- **実装に入る前に必ずユーザーの意図を確認する**
- 曖昧な点が1つでもある場合は、作業を止めて質問する
- 「たぶんこういう意味だろう」という推測で進めない

### Planモードの活用
- 可能なツール（Claude Code など）では **Planモードを使って実装計画を先に提示する**
- 計画の内容をユーザーが承認してから実装を開始する
- 計画に含める内容：
  - 変更するファイルと理由
  - 実装のアプローチ（複数ある場合は選択肢を提示）
  - 影響が出る可能性のある範囲

### 確認すべき場面
- 要件が抽象的・曖昧なとき
- 複数の実装方法が考えられるとき
- 変更範囲が広くなりそうなとき
- 既存の設計・構造に影響が出るとき
- 削除・リネームなど元に戻しにくい操作をするとき

---

## 2. コードの品質

- 既存のコードスタイル・命名規則に合わせる
- コメントは「何をしているか」ではなく「なぜそうするか」を書く
- マジックナンバーは使わず、定数として定義する
- エラーは握りつぶさず、適切に処理する
- 新しいライブラリは勝手に追加しない

---

## 3. コード量・ファイルサイズ

- 1ファイルは300行以内を目安にする（超える場合は分割を提案する）
- 関数は1つの役割だけを持たせる
- 同じ処理はコピペせず共通化する
- シンプルに書ける方法があればそちらを優先する
- 使われていないコード・変数は残さない
- ライブラリで実現できることを自前で実装しない
- **優先順位：読みやすさ ＞ 短さ**（行数削減のために読みにくくしない）

---

## 4. チーム開発のルール

- コミットメッセージは以下のフォーマットを使う
  - `feat:` 新機能
  - `fix:` バグ修正
  - `chore:` 設定・ツール変更
  - `refactor:` リファクタリング
  - `docs:` ドキュメント変更
- PRの説明には「何を・なぜ・どう変えたか」を書く
- ロジックを変更するときはテストも合わせて変更する
- `.env` やシークレット情報は絶対にコードに含めない
- 他のメンバーが触っているファイルには慎重に対応する

---

## 5. AIの暴走・ミスを防ぐ

- 削除・リネームは元に戻せないため、必ず確認してから実行する
- 自動生成したコードはレビューしてからマージする
- 方針の選択肢が複数ある場合は提示して、人間が決める
- `TODO` や `FIXME` は勝手に削除しない
- 想定外の挙動を発見した場合は作業を止めて報告する

---

## 6. ドキュメント・引き継ぎ

- コードを変更したら関連するREADMEやコメントも更新する
- 複雑な処理には「なぜこうしたか」の背景を残す
- 新しいメンバーが読んでも理解できるレベルで書く
